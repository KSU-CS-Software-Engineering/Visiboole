VisiBoole
Overview:
VisiBoole was created to be a digital logic teaching tool. It consists of both a hardware
description language (HDL) for expressing digital logic functions (components of digital
systems) and a visual simulator for that language. The interactive visual simulation can be used
to acquire an in-depth understanding of basic Boolean gates and networks. The color-coded
simulation of Boolean expressions provides a more compact display and thus better tractability
of Boolean logic (especially of larger designs) than having a schematic of a gate network
displayed. The use of the same signal name in multiple design statements implies a wire or
wires connecting those points. During simulation viewing multiple occurrences of a name
provides better tractability than viewing a web of lines showing the interconnection of sections
of a circuit diagram. During simulation the color of each named signal provides a virtual logic
probe attached to its value. Clicking on a name toggles its value while clicking on a formatted
field cycles through all combinations of values in the field making the testing and understanding
of designs natural and easy. Despite its inception as a teaching tool, VisiBoole has proven to be
a surprisingly powerful digital design tool. Its ability to support an incremental design approach
where the designer switches back and forth between design and test mode after each small
addition to the design has made the creation of complex designs remarkably easy. Further, they
are virtually bug free as soon as they are completed. The sections below introduce the HDL
features from most basic to complex. Note that all designing and testing can be accomplished
using only the assignment statement so exploring designs in simulation mode can begin after
writing a single statement. However, testing is made easier by including display statements in
the design source file.
Assignment statements:
As mentioned above it is a good idea to produce designs incrementally when using VisiBoole.
That is a good approach to learning VisiBoole as well. The VisiBoole language and simulator
have been designed to make it possible to design and test even large complex designs after
learning a single statement type - the Boolean assignment statement. It is a good idea to use it
extensively in design creation before learning more advanced features. This statement expresses
the value of a Boolean variable (a named signal) via a Boolean expression involving other
variables. The easiest to use and understand Boolean expression is a sum-of-products
expression. It uses and, or, and not (complement) operations to express an and-to-or circuit.
SOP expressions are the shortest to write and easiest to visualize during simulation. Early
versions of VisiBoole supported only SOP expressions. When a circuit involves very many
inputs a single SOP implementation usually involves too many gates or gates with too many
inputs to be practical. Using interconnected smaller circuits provides practical solutions. This is
accomplished in VisiBoole (and all HDLs) with multiple assignment statements. Each
assignment specifies a small section of the complete circuit and the signal names indicate how
these sections are interconnected. This leads into an extremely important point about HDLs in
general and VisiBoole in particular. Unlike a procedural programming language where the
statement order is usually critical, HDL statements are describing hardware and the order in
which you present segments of a design is generally unimportant. This is a key concept to
become comfortable with when learning to design with HDLs. All other HDLs have deviated
from this principle by providing order-dependent language constructs (groups of lines that
require order among themselves). VisiBoole’s HDL has been carefully created so no such
1
constructs exist. All of the statements that specify the design of any circuit, no matter complex
the design or how many inputs and outputs are involved, can be arbitrarily reordered without
affecting the functioning of the design. This is analogous to moving sections of a schematic
around the page. As long as the interconnection between the sections is maintained, the circuit
is unchanged. In VisiBoole, the interconnection between statements is implied by the names of
the signals which does not change when the statements are reordered.
As an example of a simple circuit, consider a Boolean function of three input signals named a, b,
and c, with the function being true when exactly two of these independent variables are true. A
VisiBoole assignment statement defining the gate network that will produce this function is:
f = a b ~c | a ~b c | ~a b c;
The and operator is implied whenever no other operator is connecting Boolean values. The or
operator is the vertical bar (|), with the not operator is the tilde (~). As not is a unary operator it
has the highest precedence, followed by the and operation then the or. If the function f is the
only output for the circuit to be described, the above line is the complete design! It is ready for
testing. No lines of code specifying the name of the circuit and the ports (inputs and outputs) of
the circuit are needed. Virtually all other HDLs require such a statement and also statements
specifying which port names are inputs and which are outputs. Further, if any intermediate
signals are involved, statements declaring them are required. In VisiBoole the role of all
variables are deduced by their usage. This helps immensely during an iterative design session
where intermediate variables can be used as inputs for early testing then change roles without
changing several declarations as the design progresses. As soon as the above statement is typed
into the VisiBoole IDE’s editor one can click on the RUN button and change to simulation mode
to begin interactive testing of the design. In RUN mode all variables are color coded red when
true and green when false. All independent variables start with a value of 0 (false). Binary
operators show in black. The tilde (inverter) is displayed as an overbar (a line above the variable
or parentheses enclosing the expression being inverted). Since c in the above equation starts out
false, ~c will be initially true and thus c and its overbar will be colored red.
There are two varieties of the Boolean assignment statement. The equal assignment in the
example above is use for combinational logic. Sequential logic require one-bit memory
elements known as flip-flops. Their output are known as state variables and the assignment of
them is denoted by use of the left-arrow (<=) assignment operator. During simulation their
value does not change until an edge occurs on a master clock - there is a TICK button on the
simulation screen to accomplish this. The value they are about to become, as defined by the
Boolean expression in the statement, is shown as the color of the left-arrow. Feedback of values
being assigned into the Boolean expression giving their next state is now allowed (and
frequently used). A two-bit counter design with outputs named hi and lo for its two bits is:
lo <= ~lo;
hi <= ~hi lo | hi ~lo;
There is an option on the sequential assignment operator to specify a clock signal other than the
master clock. It can be any signal in your design. To use an alternate clock you append
@clkname to the <= operator. The following code converts the above example into a counter
that counts the number of rising edges of the signal named clk:
lo <=@clk ~lo;
hi <=@clk ~hi lo | hi ~lo;

Display statements:
The display statement is just a list of signal (variable) names and/or formatted fields containing
sets of signal names ending in a semicolon. They do not contribute to the design itself and
although not required for simulation, they can play a very useful role there. They allow you to
locate input names to be toggled for testing and output names to be inspected at any and all
convenient locations throughout the design. There is no limit to the number of times a variable
can appear in a display statement or the number of display statements in which it appears. The
formatted fields allow viewing a set of values as a number in a variety of interpretations - very
helpful when testing numerical circuits. Furthermore, as you click on any formatted field
containing only independent variables and/or state variables, it will cycle through all possible
combinations of those variables making it easy to cover all possible input combinations during a
test. A formatted field has the format of %f{variable list}, where f specifies the format used to
display the list. The variable list is treated as a multibit value. The allowable formats are b
(binary), h (hexadecimal), d (2's complement converted to decimal), and u (unsigned decimal).
%u{a b c d} and %h{v[7..0]} are two examples. A cleaver unintended use of display statements
has been discovered by users. Using very short signal names one can produce rudimentary
graphical output during simulation. I believe the first such use was to show the output of a 7-
segment display by coding the following set of display statements. Note both the VisiBoole
editor and simulator use fixed-width fonts so alignment of such graphics is easy.
a a a;
f
b;
f
b;
g g g;
e
c;
e
c;
d d d;
Each set of the same signal name appears as a colored segment in simulation mode.
Exclusive-or and parentheses:
Parenthesized expressions are supported in Boolean expressions to explicitly controll the order
of operation. Thus, a product-of-sums (POS) expression indicating an or-to-and circuit could
be expressed for the function f (given earlier) as follows:
f = (a|b|c)(a|b|~c)(a|~b|c)(~a|b|c)(~a|~b|~c);
This harder to understand (for most designers) form can be interpreted as saying f will be false
when all inputs are false (first parenthesized set), none are false (last sum term), or if exactly two
are false (the middle three sum terms). This form is usually more difficult to understand. Using
the simulator to watch it work can help immensely. Renaming the function to be fsop and fpos
for the two possible forms on consecutive lines in VisiBoole and watching their operation as the
inputs are cycled through the 8 possible combinations is also enlightening. An exclusive-or
operation (^) is supported, but must be inclosed in parentheses - principally to aid in displaying
the result of the operation in simulation mode, but as a side effect it keeps the precedence rules
simple. Any number of values can be exclusive-ored together within one pair of parentheses.
The correct evaluation of a set of Boolean values exclusive-ored together is that the result is true
if an odd number of the set are true. In simulation mode parentheses are always colored to show
the Boolean value of the expression they enclose.

Vectors:
Signal names can contain letter and numbers and are case sensitive. They can be virtually any
length, but shorter is almost always better to keep the simulation screen concise. Any name that
ends in a number is referred to as an indexed variable, and can be referenced using a subscript
notation. The indices are restricted to the numbers 0 to 31 . When using subscript notation a set
of such variables can be referenced. The index has the form [beginning.step.ending]. Two
examples are v[0.2.15] and y[7..0]. When step is omitted a step of 1 is used. This allows very
compact expression for a set of values when the set involves very similar circuitry (this happens
frequently in digital systems). One example is a bus containing 8-bits that has three 8-bit
sources, x, y, and z. For this example the bus is named b7 to b0 and the three input sources are
named in a similar fashion. For this bus to take on the correct values there needs to be three
mutually-exclusive control signals indicating which value is to be selected at any given time. I
will call these sx, sy, and sz for select x, select y, and select z, respectively. The bus design then
becomes:
b[7..0] = sx x[7..0] | sy y[7..0] | sz z[7..0];
The general requirement for a vector assignment is that all vectors are the same size. Scalars are
allowed in the expression and are replicated to be the size of the vectors they are combined with.
In simulation mode the statement is expanded vertically to become eight assignment statements.
b7 = sx x7 | sy y7 | sz z7
b6 = sx x6 | sy y6 | sz z6
Continuing on to b0's assignment. This allows for very compact notation for repetitive designs.
Such notation is not only more compact, but less error prone than typing a number of similar
statements (or cutting, pasting, and modifying a statement). Using a range of indices is the most
common way to create a vector, but a vector can also be created using a concatenation operation.
This is done by enclosing a list of variables inside braces such as is done in formatted fields. A
simple example using both types of vectors is:
x[4..0] = ry y[0..4] | sc {a b z[2..0]};
which during simulation expands into:
x4 = ry y0 | sc a
x3 = ry y1 | sc b
x0 = ry y4 | sc z0
Submodules:
Submodules refer to digital designs that are used inside of a larger design. They are specified
via an instantiation statement. Such a statement is somewhat analogous to a function call in a
programming language. It can replace many assignment statements and be used multiple times
in a design. The difference between software functions and hardware submodules is that each
instantiation of a submodule places a complete copy of its hardware in the circuit, but
interconnected to a different set of signals. All such copies are functioning all the time, not
periodically as in software programs. Each submodule can have many inputs and outputs. For
example, a 4-bit adder would have nine inputs (two 4-bit numbers to add and a carry in) and five
outputs (a 4-bit sum and a carry out). Four instances of it can be connected in a way to construct
a 16-bit adder. Naming a module and explicitly listing its input and output ports is optional for
top level designs (designs that are not instantiated into any other designs). Submodules,
however, must have that information explicitly stated in a module header statement. The form
of the module header is:
4
modulename (input ports list : output ports list);
The module name must match the name of the file containing the design. The lists are comma-
separated variable lists. The variables can be scalars, named vectors, or variables in
concatenation fields. Instantiation statements in higher-level designs provide connections to the
module through variables that match the input and output lists. The designer of the submodule
controls the number and size of the ports and these are often arbitrary decisions. For example, a
full-adder has two output signals. The designer can treat these as two scalar values or a single 2-
bit vector. The three module headers below shows acceptable options. The first treats the
outputs as scalar values, the second two as a single 2-bit vector:
FullAdder(a, b, ci : co, s);
FullAdder(a, b, ci : s[1..0]);
FullAdder(a, b, ci : {co s});
Subdesigns can be placed in folders that serve as libraries of standard designs. A library
statement exists that can be placed in a subdesigns that use other subdesigns to direct the
simulator to those files. The simulator always looks in the folder containing the top-level design
first then in folders specified in library statements in the order that they occur. Library
statements have the following syntax:
#library folder_name;
The folder name can be relative to the current directory or a fully-qualified name.
Addition and Subtraction operators:
Because advanced digital systems often make use of counters or other circuits involving addition
or subtraction, those operators have been provided in VisiBoole. The beginning student should
design their own addition and subtraction circuits, then as they advance move on to using these
operators. Professional HDL provide these operators which when compiled for a reconfigurable
logic chip (FPGA) use its built-in adders. When compiled for an ASIC it will use optimized
adder/subtractor designs from a built-in library. VisiBoole restricts the use of addition (+) and
subtraction (-) operations to assignment statements involving only those two operations. In other
words, you cannot place them inside of parentheses and combine their results with Boolean
operators. You can, however, accomplish the same results by assigning them into intermediate
variables (signals) and using those names in Boolean expressions.
Constants:
VisiBoole also provides for constants (often called literals in HDLs). They have the same
format as Verilog literals except that negative values are not supported. Negative values can be
created via a subtraction operation. The radixes supported are decimal, binary, and
hexadecimal, with decimal the default. Constants can appear any place a scalar or vector value
can appear in either Boolean or arithmetic expressions. A size specification is optional unless
inside a concatenation field. Where a size is not specified the VisiBoole simulator will match
other variables in the expression. A simple example involving both subtraction and constants is
the following statement that creates a four-bit counter that counts down by 3 -- producing the
sequence: 0, 13, 10, 7, 4, 1, 14, 11, 8, 5, 2, 15, 12, 9, 6, 3, 0...
c[3..0] <= c[] - 3;
Note; the c[] is legal syntax as long as an index or indices for the variable has appeared
somewhere in the design. For empty brackets, the range used is from the largest to smallest
value that appears elsewhere.

Comparison operator:
A single comparison operator, the equal-to (==) is provided to allow a more compact expression
than writing its equivalent expression using logical operations. The two items being compared
are normally vectors and the comparison must be enclosed in parentheses. This provides an easy
way for the simulator to display the result of the comparison by coloring the enclosing
parentheses. A not-equal-to operation can be generated by placing a tilde (~) in front of the
parentheses enclosing the equal to expression. The four operations testing whether one operand
has a larger value than the other are not supported, but can be generated using the subtraction
operation. As the equal-to operator produces a binary result (true or false) it can be used with
the other logic operators in any Boolean expression in assignment statements. A complicated
example, which is not intended to have a useful purpose, but simply to demonstrate the Boolean
nature of the results of comparisons is:
b[7..0] = ~s d[7..0] | s (a[3..0] == 5) e[7..0] | s ~(f[2..0] == g[2..0]) h[7..0];
VisiBoole statements:
The VisiBoole HDL consists of six statement types:
1.
Boolean assignment statement that has one of three forms:
a. var = Boolean expression
(combinational assignment)
b. var <= Boolean expression
(sequential assignment using a master clock)
c. var <=@clk Boolean expression
(sequential assignment using an alternate clk)
Where var stands for any Boolean variable name and Boolean expression is any Boolean
equation involving Boolean variables and the operators and (implied), or (|), complement
(~), parentheses, exclusive-or (^ which must be enclosed in parentheses), and equal-to
(== which also must be enclosed in parentheses). The order of precedence in these
expressions is parentheses, complement, and, then or. The var in (a) is referred to as a
dependent variable. The vars in (b) and (c) are referred to as state variables. The clk in
(c) is the name of a variable to serve as the positive edge triggered clock for the state
variable in that piece of the design.
2.
Display statement - a list of variables and formatted fields.
3.
Instantiation statement.
4.
Arithmetic assignment statement. Same options as in the Boolean assignment statement
except that the right hand side of the assignment consists only of addition and subtraction
operations operating on vectors. It allows designs to access adder and subtraction
circuits without providing detailed designs. It is for advanced designs and allows a
concise notation for these operations which are provided in all mainstream HDLs.
5.
A module header that gives the module name and the input and output port names for the
design that follows. This allows its use as a submodule in a larger design via an
instantiation statement.
6.
A library statement to specify a directory to look for modules instantiated in the design.
